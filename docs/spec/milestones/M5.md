# M5 - Capability Governance + Sandbox Hardening

## Status

- Priority: `P0`
- Depends on: M0-M4
- Blocks: M6

## Objective

Introduce enforceable capability governance and hardened code execution suitable for enterprise multi-tenant workloads.

## In Scope

1. Capability model:
   - tenant
   - protocol
   - agent
   - skill
2. Policy engine with allow/deny, quotas, budgets.
3. Sandbox profiles:
   - `strict`
   - `balanced`
   - `extended`
4. Network/filesystem/resource controls.
5. Execution attestation.
6. OpenCode-compatible coding-machine profile.
7. Intelligent skill management:
   - skill metadata graph
   - trust/risk scoring
   - compatibility and budget-aware selection

## Out of Scope

- full zero-trust attestation infrastructure
- external IAM integration beyond local interfaces

## Required Implementation Steps

1. Define policy decision point (PDP) interfaces.
2. Enforce policy checks before every skill/sandbox execution.
3. Implement per-profile execution constraints.
4. Add attestation metadata generation and storage hooks.
5. Implement OpenCode-compatible runtime profile contract.
6. Add progressive rollout mode (`monitor`, `enforce`).
7. Implement skill metadata graph (capabilities, cost, latency, trust level, compatibility constraints).
8. Add policy-aware skill selector (allow/deny + budget + compatibility checks).
9. Add skill release channels (`stable`, `candidate`, `deprecated`) and pinning support.
10. Capture runtime skill feedback metrics (success/failure/perf/violations) for adaptive ranking.

## Required Tests

1. Unit tests:
   - policy decision logic
   - sandbox profile rule mapping
2. Security tests:
   - deny-by-default behavior
   - attempted escalation and path traversal
   - forbidden network access
3. Integration tests:
   - policy + sandbox end-to-end
   - intelligent skill selection end-to-end under policy/budget constraints
4. Stress tests:
   - concurrent coding workloads with quotas
5. Non-regression:
   - legacy safe paths still work under compatible mode

## Exit Criteria

1. No policy bypass in security test suite.
2. Sandbox constraints enforced by profile.
3. OpenCode profile executes expected workflows under constraints.
4. Skill selection is policy-safe and deterministic under declared constraints.

## LLM Delivery Checklist

1. Policy matrix examples
2. Security test outcomes
3. Coverage and residual risk summary
