# M1 - PubSub + External State

## Status

- Priority: `P0`
- Depends on: M0
- Blocks: M3-M6

## Objective

Enable production-grade asynchronous ingress and durable state with a generic PubSub adapter model (including Solace support target).

## In Scope

1. Protocol plugin `pubsub`.
2. Generic PubSub adapter interface.
3. First concrete adapters (minimum 2, one may be stubbed behind build tag if infra unavailable).
4. Explicit roadmap path for `solace` adapter.
5. External memory store support (durable conversation state).
6. Retry/ack/dlq/idempotency baseline.
7. Local runtime profiles for PubSub adapters in docker-compose.

## Out of Scope

- full control plane
- advanced policy engine

## Required Implementation Steps

1. Define adapter contract:
   - `Connect`, `Subscribe`, `Publish`, `Ack`, `Nack`, `Close`
2. Add plugin configuration shape:
   - broker type, topic(s), consumer group, retry, dlq
3. Implement message dedup strategy (`dedup_key` in meta).
4. Add failure paths:
   - nack with backoff
   - dead-letter routing
5. Add external memory backend abstraction implementation.
6. Provide `solace` adapter design contract and config scaffold (actual implementation may be staged if env is missing).
7. Add compose profiles/examples for broker adapters (at least 2 runnable locally, plus Solace config scaffold).

## Required Tests

1. Unit tests:
   - adapter contract behavior
   - dedup logic
   - retry decision logic
2. Integration tests:
   - end-to-end pubsub ingress -> orchestrator -> reply
   - restart recovery with persistent memory
3. Failure injection:
   - duplicate delivery
   - transient broker failure
   - ack timeout
4. Non-regression:
   - `http`, `telegram`, `webhook` unchanged
5. Runtime profile tests:
   - compose profile boot for each supported local broker

## Exit Criteria

1. At-least-once semantics verified.
2. Idempotent processing proven by tests.
3. Recovery after restart works with persistent session/history.
4. Solace adapter path documented and config-valid.
5. PubSub adapter profiles are executable from local startup scripts.

## LLM Delivery Checklist

1. Adapter interface and implementations list
2. Test matrix by broker/failure case
3. Coverage report
4. Operational caveats and deployment notes
