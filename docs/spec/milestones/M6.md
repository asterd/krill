# M6 - Enterprise Control Plane + Operations Completion

## Status

- Priority: `P1`
- Depends on: M0-M5
- Blocks: none

## Objective

Provide centralized enterprise operations for protocols, agents, capabilities, planning policy, scheduling, sessions, observability, and deployment/runtime packaging so that the system can be safely managed as a product platform.

This milestone must close the operational gap for:

1. managed scheduling and recurring activity operations
2. managed multi-agent orchestration and policy rollout
3. managed custom capability lifecycle and secrets/policy changes
4. local/cluster runtime parity needed for hosted operational assets and enterprise rollout

## Use-Case Coverage Target

- Primary:
  - use case 3 (Scheduling activities)
- Enabling:
  - use case 5 (managed multi-agent operations)
  - use case 6 (managed deployment/runtime path)
  - use case 2 (managed capability lifecycle and secrets ops)

## In Scope

1. Control plane API:
   - protocol management
   - agent profile management
   - capability lifecycle management
   - policy/capability management
   - planner configuration and diagnostics
   - plan/session inspection
   - scheduler management
   - session diagnostics
   - artifact/deployment run inspection
   - external action approval status
2. Tenant and RBAC model.
3. Secrets rotation and audit retention policies.
4. Health, readiness, and diagnostics endpoints.
5. Operational runbook.
6. Helm chart packaging and cluster startup scripts.
7. Installation guides for Kubernetes and OpenShift.

## Out of Scope

- full-featured GUI (API-first scope)
- enterprise billing

## Required Implementation Steps

1. Define control plane authn/authz contracts.
2. Add CRUD APIs for manageable entities.
3. Add planner-facing APIs for:
   - capability metadata management
   - policy bindings
   - planner mode/configuration
   - plan inspection/replay metadata access
4. Add immutable audit events for control operations.
5. Add safe rollout semantics:
   - dry run
   - staged apply
   - rollback
6. Add operator diagnostics for incident response, including plan traces, policy decisions, and capability selection rationale.
7. Add APIs or diagnostics endpoints to distinguish:
   - direct-loop execution
   - planner-based execution
   - cooperative workflow execution
8. Deliver Helm chart under `deploy/charts/krill` with values overlays (`dev`, `minikube`, `prod`).
9. Add scripts:
   - `deploy/scripts/k8s/up-minikube.sh`
   - `deploy/scripts/k8s/install.sh`
   - `deploy/scripts/k8s/uninstall.sh`
10. Add installation docs:
   - `deploy/docs/k8s.md`
   - `deploy/docs/openshift.md`
11. Ensure cluster path deploys the same logical services as local compose path.
12. Ensure control-plane managed planner/policy/capability changes can be rolled out without process restart where feasible.
13. Add scheduler operations sufficient for use case 3:
   - create/update/pause/resume/cancel schedule
   - inspect schedule history and next run
   - inspect schedule-triggered plan outcomes
14. Add artifact/deployment operations sufficient for use case 6:
   - inspect generated artifacts
   - inspect external action audit trail
   - staged rollout and rollback metadata
15. Add cooperative orchestration diagnostics sufficient for use case 5:
   - inspect handoff chain
   - inspect capability denials
   - inspect escalation/approval waits
16. Ensure secrets and capability changes can be rotated/applied without invalidating active runtime sessions where feasible.

## Required Tests

1. Unit tests:
   - RBAC and authz logic
   - API validation and defaults
   - planner config validation
   - capability metadata validation
2. Integration tests:
   - end-to-end control-plane operations to data-plane effect
   - policy/capability updates reflected in planner selection behavior
   - plan inspection endpoints reflect runtime state and audit trail
   - schedule CRUD to runtime effect
   - secrets rotation to capability effect
   - artifact/deployment inspection endpoints
   - cooperative workflow diagnostics reflect runtime state
3. Security tests:
   - unauthorized operation rejection
   - audit tamper checks
4. Non-regression:
   - data plane keeps functioning with control plane disabled/failing
5. Deployment tests:
   - Helm lint/template validation
   - mini-kube install/uninstall smoke
   - OpenShift compatibility checks (manifest/security context)

## Exit Criteria

1. Core operational workflows manageable without process restart.
2. RBAC and audit verified by tests.
3. Operators can inspect plan decisions, capability selection, and policy denials without direct process access.
4. Runbook complete and validated in a simulated incident drill.
5. Helm + mini-kube flow works from one script.
6. Kubernetes and OpenShift install docs are complete and validated.
7. Scheduling workflows required by use case 3 are operable end-to-end without direct process access.
8. Multi-agent and external-action workflows required by use cases 5 and 6 are observable and governable through control-plane APIs.
9. Capability lifecycle needed by use case 2 can be managed, audited, and rolled out safely.
