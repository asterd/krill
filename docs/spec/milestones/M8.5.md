# M8.5 - Performance Hardening + Latency Budgets + Runtime Tightening

## Status

- Priority: `P0`
- Depends on: M0-M8
- Blocks: none

## Objective

Take the advanced execution intelligence introduced through M8 and harden it into a production-grade ultra-light runtime with explicit latency, throughput, and overhead budgets.

This milestone exists to ensure Krill does not merely support graph planning, capability retrieval, and nested execution, but does so within a tightly controlled performance envelope suitable for enterprise-grade interactive and operational workloads.

M8.5 is the "make it serious" milestone for runtime efficiency.

## Use-Case Coverage Target

- Primary:
  - use case 1 (OpenClaw-like agent-loop and chat capabilities) under strict interactive latency budgets
  - use case 2 (Invoke custom skills) under bounded execution overhead
  - use case 5 (Orchestrazione multi-agent autonoma) under bounded graph/runtime cost
- Enabling:
  - use case 3 (Scheduling activities) under controlled background concurrency
  - use case 6 (Creazione asset operativi) under bounded operational execution overhead

## Design Constraints

1. Performance first:
   - every new advanced planner/runtime path must justify its overhead
   - planner intelligence must not degrade direct-loop performance
2. Lightweight by default:
   - avoid mandatory heavyweight infrastructure
   - keep fast local mode first-class
3. Deterministic degradation:
   - when budgets are exceeded, Krill must simplify, prune, or fall back predictably
4. Observability must remain useful but cheap:
   - no excessive tracing/logging overhead on hot paths

## In Scope

1. Explicit runtime SLO/SLA-style engineering budgets:
   - planner build latency
   - retrieval latency
   - direct-loop overhead delta
   - graph execution overhead
   - nested execution overhead
2. Runtime tightening:
   - hot-path allocations review
   - reduced prompt/capability context serialization overhead
   - cache strategy hardening
   - bounded concurrency tuning
3. Planner and retrieval efficiency controls:
   - adaptive top-k capability retrieval
   - plan complexity limits
   - graph width/depth caps
   - prompt compaction and reuse
4. Parallel execution efficiency:
   - worker-pool tuning
   - branch scheduling fairness
   - join-node backpressure handling
5. Cost-aware execution:
   - token budget optimization
   - capability cost-aware pruning
   - plan short-circuiting on sufficient result quality
6. Runtime safeguards:
   - overload shedding
   - bounded queue growth
   - cancellation propagation
   - deadline enforcement
7. Benchmark and profiling suite:
   - microbenchmarks
   - concurrency benchmarks
   - regression benchmarks
   - representative end-to-end performance scenarios

## Out of Scope

- new protocols or new product features
- new control-plane feature areas
- GUI or analytics dashboards
- large-scale distributed scheduler rearchitecture

## Required Implementation Steps

1. Define explicit performance budgets for key paths:
   - direct loop routing overhead
   - planner routing overhead
   - plan build latency
   - retrieval latency
   - graph executor scheduling latency
2. Add configuration and defaults for:
   - max graph width
   - max graph depth
   - max retrieval top-k
   - max nested depth
   - planner timeout budget
   - retrieval timeout budget
3. Add planner complexity guards:
   - deny or simplify overly large plans
   - collapse excessive branch fan-out
   - reject pathological nested execution trees
4. Add prompt compaction and planner context reuse:
   - summarized capability metadata payloads
   - reusable plan skeletons
   - stable serialization for cache hits
5. Add adaptive retrieval controls:
   - top-k shrink on simple requests
   - top-k expansion only when confidence is low
   - metadata-first filter before semantic scoring
6. Add executor scheduling improvements:
   - ready-queue efficiency
   - bounded goroutine growth
   - per-plan concurrency caps
   - fairness across concurrent plans
7. Add backpressure and overload behavior:
   - queue rejection or downgrade policy
   - timeout-aware branch cancellation
   - safe fallback from graph -> linear -> direct where appropriate
8. Add runtime memory discipline:
   - cap temporary artifact growth
   - bound in-memory plan cache
   - bound retrieval cache
   - bound audit payload size where safe
9. Add observability tuning:
   - disable high-cost telemetry on hot paths by default
   - support sampled detailed diagnostics
   - preserve correctness of incident traces under sampling
10. Add a benchmark suite with representative scenarios:
   - short conversational request
   - single-step governed capability request
   - graph plan with parallel branches
   - nested skill invocation
   - scheduler-triggered graph execution
11. Add profiling workflow and docs for:
   - CPU hotspots
   - allocation hotspots
   - lock contention
   - prompt/token overhead
12. Add fail-fast paths when budgets are at risk:
   - prune branches
   - reduce candidate set
   - skip optional synthesis passes
   - fall back to cheaper plan mode
13. Add config overlays for dev/minikube/prodlike with tuned defaults for performance-sensitive planner settings.

## Required Tests

1. Unit tests:
   - budget enforcement
   - plan simplification and pruning
   - adaptive top-k logic
   - overload downgrade behavior
   - timeout and cancellation propagation
2. Integration tests:
   - graph plan remains within configured bounds
   - direct-loop latency path unaffected by planner-heavy features
   - overload behavior remains deterministic and safe
   - nested execution fails safely under depth/width limits
3. Benchmark tests:
   - planner route decision benchmark
   - plan build benchmark
   - retrieval benchmark
   - graph executor benchmark
   - nested skill invocation benchmark
4. Failure-injection tests:
   - retrieval timeout
   - planner timeout
   - branch overload
   - cache stampede scenario
   - backpressure under concurrent plans
5. Non-regression:
   - M8 graph planner correctness preserved under optimizations
   - M5-M7 functionality preserved under tighter runtime guards

## Exit Criteria

1. Direct-loop requests remain on the low-overhead path with no material regression against the pre-M8 baseline.
2. Planner and retrieval paths operate within declared latency budgets under representative test scenarios.
3. Graph execution remains bounded in width, depth, and concurrency under configuration and tests.
4. Nested execution remains safe and bounded under stress and failure scenarios.
5. Adaptive retrieval reduces unnecessary context while preserving capability selection correctness.
6. Overload and timeout behavior are deterministic, auditable, and verified by tests.
7. Benchmarks and failure-injection evidence show Krill remains ultra-light and ultra-fast despite advanced planning capabilities.
8. The implementation demonstrates that advanced execution intelligence can coexist with strict runtime efficiency goals.
