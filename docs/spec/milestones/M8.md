# M8 - Graph Planner + Capability Retrieval + Composable Skill Runtime

## Status

- Priority: `P0`
- Depends on: M0-M7
- Blocks: none

## Objective

Close the final execution-intelligence gap so Krill can behave like a truly efficient agent runtime for complex work without becoming a heavyweight orchestration platform.

This milestone must introduce:

1. graph-shaped planning for non-trivial requests
2. intelligent low-overhead capability/skill retrieval with minimal context expansion
3. first-class nested and composable skill execution
4. token/cost/latency-aware planning decisions that preserve Krill's lightweight runtime character

M8 exists to make Krill operationally and product-wise complete for advanced agent execution patterns without turning the runtime into a bloated workflow engine. The design target is not "maximum abstraction", but "maximum useful intelligence per unit of runtime overhead".

## Use-Case Coverage Target

- Primary:
  - use case 1 (OpenClaw-like agent-loop and chat capabilities) for complex planning/research/coding paths
  - use case 2 (Invoke custom skills) for intelligent capability discovery and composition
  - use case 5 (Orchestrazione multi-agent autonoma) for graph-based autonomous execution
- Enabling:
  - use case 3 (Scheduling activities) for graph-capable scheduled execution
  - use case 6 (Creazione asset operativi) for multi-step operational/deployment plans

## Design Constraints

1. The runtime must remain ultra-light:
   - avoid introducing a heavyweight DAG/workflow engine by default
   - keep in-process graph execution as the primary path
   - avoid mandatory external control/state services for local execution
2. The planner must remain ultra-fast:
   - no full-catalog prompt stuffing on every turn
   - no unnecessary plan generation for simple requests
   - bounded planning rounds
3. The planner must remain ultra-efficient:
   - optimize for token cost, latency, and memory footprint
   - prefer direct loop or single-step execution when sufficient
   - retrieve only the minimum capability set needed for the current request
4. Backward compatibility remains mandatory:
   - existing direct loop, cooperative mode, scheduler mode, and skill registry behavior must not break

## In Scope

1. Graph planner runtime:
   - explicit DAG-style plan model
   - node dependencies
   - serial and parallel step execution
   - join/merge nodes
   - bounded graph depth/width
2. Plan optimizer:
   - decide between direct loop, linear plan, or graph plan
   - token/cost/latency-aware execution path selection
   - plan simplification and pruning
   - bounded re-planning
3. Intelligent capability retrieval:
   - semantic capability discovery
   - metadata-aware capability retrieval
   - tag + policy + trust + compatibility + semantic ranking
   - minimal context packing for planner/executor prompts
4. Capability memory/index:
   - low-overhead capability metadata index
   - optional embedding-backed retrieval
   - fallback lexical/metadata retrieval path
   - cache and invalidation strategy
5. Composable skill runtime:
   - skill calling another skill through runtime contracts
   - skill calling a capability through runtime contracts
   - parent/child trace, budget, and policy propagation
   - recursion guard and call-depth limits
6. Artifact and branch merge handling:
   - structured outputs from graph branches
   - merge strategy for artifacts and intermediate results
   - conflict/failure reporting
7. Scheduling and cooperative execution uplift:
   - scheduler may emit graph plans when appropriate
   - cooperative workflows may use graph branches rather than only linear role pipelines
8. Plan reuse and caching:
   - reusable plan skeletons for recurring intents
   - request canonicalization for near-duplicate requests
   - bounded cache with hit/miss metrics

## Out of Scope

- external distributed workflow engine as a hard dependency
- unbounded autonomous self-modifying planning loops
- full marketplace/recommender system for third-party skills
- mandatory vector database dependency for local mode
- GUI control-plane implementation
- A2UI

## Required Implementation Steps

1. Extend the `Plan` model to support graph semantics:
   - node ids
   - edge/dependency set
   - execution groups
   - merge nodes
   - branch-local artifact outputs
2. Introduce a `PlanModeSelector` that can choose:
   - direct loop
   - linear plan
   - graph plan
3. Add strict heuristics to avoid planner overuse:
   - simple requests stay on direct loop
   - single-capability requests avoid graph planning
   - graph planning only for meaningful fan-out/fan-in or branch-dependent work
4. Implement a lightweight in-process graph executor with:
   - ready queue
   - bounded worker pool
   - deterministic scheduling policy
   - dependency tracking
   - branch cancellation on fatal policy failures where required
5. Add branch merge contracts for:
   - text synthesis
   - structured JSON merge
   - artifact collection
   - partial-failure summarization
6. Introduce a `CapabilityRetriever` abstraction with:
   - semantic retrieval
   - metadata filtering
   - policy-aware candidate reduction
   - deterministic ordering
7. Add a low-overhead capability index:
   - capability name
   - description
   - tags
   - auth/network/runtime requirements
   - trust/risk/cost/latency metadata
   - optional embedding/vector payload
8. Ensure retrieval is incremental:
   - do not inject the full capability catalog into planning context by default
   - inject only top-k relevant capabilities
   - make top-k configurable and bounded
9. Add multilingual capability retrieval support sufficient for:
   - Italian
   - English
   - paraphrased intent forms
10. Introduce first-class nested execution contracts:
   - `SkillRuntime.InvokeSkill(...)`
   - `SkillRuntime.InvokeCapability(...)`
   - child context propagation
   - trace/budget inheritance
   - policy decision re-check before child execution
11. Add nested execution safety controls:
   - max nested depth
   - max child fan-out
   - cycle detection
   - recursion denial reason reporting
12. Add a structured `ExecutionFrame` or equivalent runtime model that records:
   - parent node
   - child node
   - origin capability/skill
   - inherited budget
   - approval and escalation state
13. Add plan optimization hooks:
   - collapse trivial nodes
   - fuse consecutive low-risk steps where safe
   - reuse previous validated plan skeletons
   - short-circuit on sufficient results
14. Add a planner token-efficiency strategy:
   - summarized capability metadata
   - request canonicalization
   - retrieved candidate caps only
   - plan skeleton cache
15. Add plan execution metrics:
   - plan build latency
   - retrieval latency
   - graph width/depth
   - parallel branch count
   - cache hit/miss
   - token/cost saved by retrieval/pruning when estimable
16. Extend scheduler integration so a scheduled run can:
   - use linear plan
   - use graph plan
   - emit branch-aware audit entries
17. Extend cooperative orchestration so a multi-agent workflow may:
   - execute serially
   - branch to multiple specialists
   - merge back into a synthesizer
   - remain bounded and policy-safe
18. Add explicit fallback behavior for environments without embedding services:
   - metadata-only retrieval
   - lexical ranking
   - deterministic reduced candidate selection
19. Ensure all new planner/retrieval behavior is inspectable through plan/audit state so M6 control-plane patterns remain compatible.
20. Keep local and cluster runtime parity:
   - no feature should work only in one path
   - graph planner/retrieval runtime config must support local compose and Helm overlays

## Required Tests

1. Unit tests:
   - plan mode selector
   - graph dependency validation
   - ready-queue scheduling
   - merge strategy logic
   - retrieval ranking and candidate reduction
   - multilingual/paraphrase capability retrieval
   - nested execution frame propagation
   - recursion guard and cycle detection
   - plan skeleton cache behavior
2. Integration tests:
   - conversational request -> graph plan -> merged final answer
   - graph plan with parallel branches under bounded worker pool
   - scheduler trigger -> graph plan -> audited outcome
   - cooperative multi-agent branch -> synthesis merge
   - capability retrieval reduces planning context without breaking tool correctness
   - nested skill -> child skill execution with inherited policy/budget
   - nested skill -> child capability execution with trace continuity
3. Performance tests:
   - planner latency vs direct loop baseline
   - retrieval overhead budget
   - graph executor parallelism under bounded concurrency
   - token usage comparison:
     - full-catalog planner context
     - retrieved top-k capability context
4. Failure-injection tests:
   - branch failure with partial merge result
   - child skill denial due to policy
   - recursive/nested execution limit breach
   - retrieval backend unavailable -> metadata fallback path
   - stale plan skeleton rejected and rebuilt
5. Non-regression:
   - simple direct-loop conversations remain direct
   - existing lazy skill loading remains compatible
   - M5 plan auditing and M6 control-plane assumptions remain valid

## Exit Criteria

1. Krill can build and execute graph-shaped plans for appropriate non-trivial requests without requiring a heavyweight external workflow engine.
2. Parallel step execution is supported where dependencies allow it, and remains bounded, auditable, and deterministic.
3. Capability retrieval reduces planner context size materially compared with full-catalog planning, with measurable token or prompt-size savings.
4. Retrieval remains correct for multilingual and paraphrased requests relevant to use cases 1 and 2.
5. Nested skill/capability execution works as a first-class runtime feature with inherited policy, trace, and budget controls.
6. Recursive or cyclic nested execution is denied safely and audibly.
7. Cooperative multi-agent execution can branch and merge through the graph planner without undefined behavior.
8. Scheduler-triggered executions can use graph plans when needed and still satisfy governance/audit requirements.
9. Direct-loop requests remain on the low-overhead path and do not regress in latency or compatibility.
10. The implementation demonstrates that advanced planning intelligence was added without violating the product goal of remaining ultra-light, ultra-fast, and ultra-performant.
