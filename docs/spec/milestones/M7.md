# M7 - Baseline Product Completion + Direct Skill and Operational Runtime

## Status

- Priority: `P0`
- Depends on: M0-M6
- Blocks: none

## Objective

Close the remaining product and runtime gaps needed to satisfy all remaining use cases except A2UI:

1. OpenClaw-like conversational agent capabilities across the channels already present in core runtime.
2. First-class custom skill invocation for enterprise integrations and complex task execution.
3. Product-grade scheduled and operational flows exposed through APIs and existing conversational surfaces.
4. Operational asset creation and hosted deployment workflows.

This milestone is complete only when Krill can be presented as a production-ready baseline conversational agent and operational automation platform on top of the runtime already delivered through M0-M6, not yet as a channel-complete Skill OS.

Dedicated Slack/Discord/WhatsApp adapters, GUI-grade HITL flows, large-catalog skill retrieval, and arbitrary graph planning are explicitly deferred beyond M7 and must not be silently assumed complete here.

## Use-Case Coverage Target

- Baseline end-to-end product completion:
  - use case 1 (OpenClaw-like agent-loop and chat capabilities)
  - use case 2 (Invoke custom skills)
  - use case 6 (Creazione asset operativi)
- End-to-end productization on top of M5/M6 foundations for bounded governed flows:
  - use case 3 (Scheduling activities)
  - use case 5 (Orchestrazione multi-agent autonoma)
- Deferred to M7.5 and M8:
  - dedicated Slack/Discord/WhatsApp adapters
  - first-class human-in-the-loop conversational approval UX
  - arbitrary graph planning
  - semantic capability retrieval
  - first-class nested skill/capability composition

## In Scope

1. Baseline conversational productization:
   - `http`
   - `telegram`
   - existing generic `webhook` fallback where appropriate
2. Multimodal input and output path:
   - text
   - audio/voice
   - images
   - files/documents
3. OpenClaw-like task capability baseline:
   - weather and simple web tasks
   - external system verification
   - document summarization
   - internet research
   - code generation and execution
4. Native skill invocation surfaces:
   - conversational tool usage
   - direct API invocation
   - workflow step invocation
5. Custom skill integration model:
   - database access
   - authenticated API access
   - complex multi-step execution
   - code example emission when relevant
6. Skill trust and runtime safety:
   - secrets handling
   - policy enforcement
   - audit trail
   - execution metadata
7. Reply routing and delivery guarantees for interactive channels.
8. Product-grade docs and example configurations for the supported channels and skill patterns.
9. Product-grade scheduled interaction surfaces:
   - conversational schedule creation/update
   - API schedule invocation and inspection
   - notification delivery for scheduled outcomes
10. Product-grade operational asset lifecycle:
   - artifact generation
   - deployment/hosting capability execution
   - runtime status and rollback hooks
11. Human-in-the-loop operational flows over current surfaces:
   - approval requests over API
   - follow-up data collection over existing channels
   - escalation notifications for autonomous workflows over existing channels

## Out of Scope

- scheduling and cron lifecycle beyond direct invocation surfaces
- A2UI and UI-generation protocol support
- marketplace billing or third-party commercial packaging

## Required Implementation Steps

1. Native protocol plugins for Slack/Discord/WhatsApp are not required in M7:
   - remain deferred to M7.5
2. Upgrade the existing `webhook` plugin role:
   - keep it as generic ingress fallback
   - stop treating it as sufficient evidence of Slack/Discord/WhatsApp parity
3. Implement bidirectional delivery contracts for all conversational protocols:
   - inbound normalization
   - outbound reply delivery
   - retry and idempotency hooks where provider semantics require them
4. Add multimodal envelope support end-to-end:
   - ingest attachments into canonical envelope
   - persist attachment references with message history
   - pass multimodal content to LLM backends that support it
   - provide degradation path for text-only backends
5. Add voice path:
   - speech-to-text for inbound audio
   - optional text-to-speech for outbound replies
   - transcript persistence and provenance metadata
6. Add document ingestion pipeline:
   - plain text
   - PDF
   - common office/document formats if supported by chosen extractor
   - chunking and summarization path
7. Add built-in product capability skills:
   - weather provider skill
   - web/search skill using approved provider abstraction
   - document summarize/extract skill
   - system health/API probe skill
8. Add a direct skill invocation API:
   - `POST /v1/skills/{name}:invoke`
   - optional async mode
   - structured input validation
   - trace/audit correlation
9. Add skill execution contracts for enterprise connectors:
   - database connection profile
   - authenticated HTTP/API profile
   - multi-step skill chaining contract
10. Add secrets resolution model for skills:
   - env-backed secrets
   - config references
   - redact-on-log behavior
   - explicit deny for missing required secrets
11. Extend skill metadata model:
   - capabilities
   - auth requirements
   - network requirements
   - data sensitivity
   - expected latency/cost
12. Ensure policy enforcement occurs before every direct or conversational skill invocation.
13. Extend `code_exec` workflow guidance:
   - support returning runnable code examples as artifact payloads
   - distinguish "example only" from "executed result"
14. Add user-visible response contracts for tool-backed answers:
   - source attribution where applicable
   - execution summary for skill results
   - partial-failure explanation
15. Add provider abstraction for research/search/weather so implementations are replaceable without protocol changes.
16. Implement conversation-quality regression suite:
   - tool selection correctness
   - multimodal ingestion correctness
   - channel reply correctness
17. Update deployment examples and docs:
   - HTTP/Telegram/webhook setup guides
   - custom skill recipes for DB and authenticated API calls
   - operational security guidance
18. Add direct and conversational surfaces for use case 3:
   - create/update/pause schedules through API
   - schedule requests through supported current conversational channels where possible
   - render schedule outcomes and failures back to users
19. Add product-grade flows for use case 5:
   - user-visible autonomous workflow summaries
   - escalation prompts when the workflow needs approval or data
   - final synthesis and audit trail delivery over channels/API
   - bounded cooperative/linear execution is sufficient for M7; arbitrary graph expansion is not required here
20. Add product-grade flows for use case 6:
   - asset/deployment skill recipes
   - hosting/deployment provider abstraction
   - artifact packaging and result links
   - failure explanation and rollback guidance
21. Add acceptance suites directly derived from `USE_CASES.md` excluding A2UI.

2. Integration tests:

## Exit Criteria

1. A user can converse with Krill over Telegram, HTTP, and supported webhook-based ingress paths with correct bidirectional or provider-compatible message delivery.
2. Text, voice, image, and document inputs are accepted and processed through an auditable multimodal path where the configured provider stack supports them.
3. Krill can reliably complete the concrete task classes named in use case 1:
   - weather
   - system verification
   - document summarization
   - internet research
   - code development support
4. A custom skill can be invoked both:
   - by the conversational agent through tool calling
   - directly through a stable API surface
5. Custom skills can safely perform the concrete task classes named in use case 2:
   - database query
   - authenticated API call
   - complex multi-step activity execution
   - code example generation
6. Secrets, policy decisions, and execution outcomes are auditable and verified by tests.
7. Use case 3 is demonstrably satisfied end-to-end through API and, where applicable, conversational surfaces.
8. Use case 5 is demonstrably satisfied end-to-end for bounded cooperative/autonomous workflow execution, escalation, and user-visible outcomes.
9. Use case 6 is demonstrably satisfied end-to-end with asset generation and hosting/deployment workflows.
10.
