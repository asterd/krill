# M7.5 - Channel Completion + HITL Product Flows + Operational Connectors

## Status

- Priority: `P0`
- Depends on: M0-M7
- Blocks: M8

## Objective

Close the product-platform gap between "baseline product runtime" and "Skill OS-style operational surface" by delivering:

1. first-class protocol adapters for enterprise chat/task ingress
2. human-in-the-loop flows that can pause, request approval/data, and resume
3. operational connector surfaces that make enterprise skills usable as product features, not just runtime primitives

This milestone exists because the jump from M7 to a true Skill OS is not mainly about better planning. It is first about better surfaces: dedicated channels, approval UX, escalation delivery, and resumable operator workflows.

## Use-Case Coverage Target

- Primary:
  - use case 1 (OpenClaw-like agent-loop and chat capabilities)
  - use case 2 (Invoke custom skills)
  - use case 5 (Orchestrazione multi-agent autonoma)
- Enabling:
  - use case 3 (Scheduling activities)
  - use case 6 (Creazione asset operativi)

## In Scope

1. Dedicated protocol adapters:
   - `slack`
   - `discord`
   - `whatsapp`
   - optional `email` ingress/notification adapter if kept lightweight
2. Channel-specific delivery semantics:
   - reply mapping
   - retry/idempotency
   - provider callback verification
   - threaded conversation correlation where available
3. HITL runtime model:
   - explicit `waiting_for_approval`
   - explicit `waiting_for_input`
   - resume token or request correlation
   - timeout/escalation path
4. HITL delivery surfaces:
   - control-plane/API approval
   - conversational approval/rejection
   - follow-up question collection
   - escalation notifications
5. Channel-visible workflow state:
   - `in_progress`
   - `needs_approval`
   - `needs_input`
   - `completed`
   - `failed`
6. Operational connector productization:
   - DB/API connectors exposed through stable skill contracts
   - notification connectors
   - external action/deployment result delivery contracts
7. Approval and escalation audit continuity across protocol, planner, control plane, and schedule execution.

## Out of Scope

- arbitrary DAG/graph planning
- semantic/embedding capability retrieval
- nested skill runtime
- full GUI control plane
- marketplace/recommender ecosystem

## Required Implementation Steps

1. Add native protocol plugins for:
   - `plugins/protocol/slack`
   - `plugins/protocol/discord`
   - `plugins/protocol/whatsapp`
2. If `email` is included, keep it minimal:
   - inbound task normalization
   - outbound notification delivery
   - no heavy mailbox-management scope
3. Introduce a stable workflow wait-state contract shared across runtime components:
   - request id
   - wait reason
   - requested actor
   - expiry
   - resume action
4. Extend plan and workflow snapshots so paused/waiting executions are inspectable and resumable through API and channels.
5. Add a resume path from:
   - control-plane approval
   - channel reply/interaction
   - API callback
6. Add channel-facing approval UX contracts:
   - approve
   - reject
   - provide missing data
   - request more details
7. Add notification capability contracts for:
   - send approval request
   - send escalation
   - send final summary
   - send partial-failure summary
8. Add enterprise connector recipes and contracts:
   - authenticated HTTP/API action
   - database query action
   - operational/deployment action result reporting
9. Add schedule integration for HITL:
   - scheduled flow can pause for approval
   - scheduled flow can resume after approval/input
   - schedule audit reflects waiting/resume path
10. Add user-facing and operator-facing examples for:
   - autonomous workflow requiring approval
   - custom skill requiring missing secret/data
   - scheduled operational workflow requiring operator confirmation

## Required Tests

1. Unit tests:
   - adapter payload normalization
   - HITL state transitions
   - resume token/correlation validation
   - notification capability contract behavior
2. Integration tests:
   - `slack` end-to-end chat with replies
   - `discord` end-to-end chat with replies
   - `whatsapp` end-to-end chat with replies
   - approval request -> operator/user response -> workflow resume
   - missing data request -> user response -> workflow resume
   - scheduled workflow -> approval wait -> resume -> final outcome
3. Security tests:
   - forged callback rejection
   - unauthorized approval/reject action rejection
   - cross-thread/cross-request resume denial
4. Non-regression:
   - HTTP/Telegram/webhook flows from M7 remain compatible
   - control-plane approval APIs remain valid

## Exit Criteria

1. Krill has first-class dedicated adapters for Slack, Discord, and WhatsApp, not only webhook-based approximation.
2. Approval and missing-input flows can pause and resume execution safely across channels and APIs.
3. Multi-agent and scheduled workflows can escalate to humans and continue after approval/input without undefined runtime behavior.
4. Enterprise connector actions can produce product-grade user/operator notifications rather than only internal audit artifacts.
5. The platform can credibly claim channel-complete and HITL-complete baseline coverage for use cases 1, 2, 3, and 5.
