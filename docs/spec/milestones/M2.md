# M2 - OTEL Deep Observability

## Status

- Priority: `P0`
- Depends on: M0, M1
- Blocks: M3-M6

## Objective

Provide enterprise OTEL observability with detailed signals and bounded overhead.

## In Scope

1. OTLP exporters for traces/metrics/log correlation.
2. Span instrumentation across protocol, bus, orchestration, LLM, skill, memory, sandbox.
3. Metrics and labels standardization.
4. Profile levels: `off|minimal|standard|debug`.
5. Benchmarking overhead and sampling strategies.

## Out of Scope

- custom dashboard UI
- full SIEM integrations beyond OTLP compatibility

## Required Implementation Steps

1. Introduce OTEL config schema with sane defaults.
2. Instrument required spans and preserve context propagation.
3. Add required metrics:
   - `krill.active_loops`
   - `krill.inbound_queue_depth`
   - `krill.skill.activations_total`
   - `krill.memory.ops_total`
   - `krill.memory.bytes`
   - `krill.sandbox.exec_duration_ms`
   - `krill.agent.handoff_total`
   - `krill.scheduler.trigger_total` (future-proof for M4)
   - `krill.session.resume_total` (future-proof for M4)
4. Implement sampling/verbosity per profile.
5. Add benchmark harness: baseline vs profile overhead.

## Required Tests

1. Unit tests:
   - trace context propagation
   - metric emission guards
2. Integration tests:
   - end-to-end trace correlation from ingress to reply
3. Benchmarks:
   - `minimal` and `standard` overhead assertions
4. Non-regression:
   - behavior unchanged when OTEL is `off`

## Exit Criteria

1. Observability profile `minimal` <= 3% CPU overhead.
2. Profile `standard` <= 8% CPU overhead.
3. Full trace correlation validated.
4. Mandatory metrics visible in exporter pipeline.

## LLM Delivery Checklist

1. Instrumentation map (where each span/metric is emitted)
2. Overhead benchmark result
3. Coverage and regression summary
