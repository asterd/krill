# M5 - Capability Governance + Planning Runtime

## Status

- Priority: `P0`
- Depends on: M0-M4.5
- Blocks: M6

## Objective

Introduce enforceable capability governance, hardened code execution, and a lightweight first-class planning/runtime decision layer that closes the core execution gaps for:

1. custom skill invocation with enterprise safety guarantees
2. autonomous multi-agent orchestration with explicit plans and bounded budgets
3. complex scheduled activity execution as auditable plans
4. asset creation and external operational actions under policy

This milestone must make Krill capable of deciding, selecting, and safely executing complex work. It does not yet need to be fully product-complete on channels or multimodality, but it must remove the architecture gaps that would otherwise prevent full coverage of use cases 2, 3, 5, and 6.

## Use-Case Coverage Target

- Primary:
  - use case 2 (Invoke custom skills)
  - use case 5 (Orchestrazione multi-agent autonoma)
- Foundational:
  - use case 3 (Scheduling activities)
  - use case 6 (Creazione asset operativi)
- Partial support only:
  - use case 1 (planning, research/tool orchestration path only; not full product parity)

## In Scope

1. Capability model:
   - tenant
   - protocol
   - agent
   - skill
   - generic capability abstraction (`skill|mcp|code|backend_action|agent_action`)
   - deployment/asset action capability abstraction
   - notification/escalation capability abstraction
2. Policy engine with allow/deny, quotas, budgets.
3. Sandbox profiles:
   - `strict`
   - `balanced`
   - `extended`
4. Network/filesystem/resource controls.
5. Execution attestation.
6. OpenCode-compatible coding-machine profile.
7. Lightweight planning primitives:
   - `IntentResolver`
   - explicit `Plan`
   - `CapabilitySelector`
   - `PlanExecutor`
   - `ExecutionPolicyGate`
   - `EscalationResolver`
   - `ArtifactCollector`
8. Intelligent capability management:
   - capability metadata graph
   - trust/risk scoring
   - compatibility and budget-aware selection
   - deterministic fallback ordering
9. Direct support for task classes needed by remaining use cases:
   - database/API/action capabilities routed through capability abstraction
   - scheduled work routed through plan execution instead of ad-hoc trigger execution
   - autonomous cooperative-agent plans with handoff guardrails
   - asset/deployment actions represented as policy-governed capabilities

## Out of Scope

- full zero-trust attestation infrastructure
- external IAM integration beyond local interfaces
- full autonomous workflow compiler or heavyweight DAG engine
- replacing the existing direct conversational loop for simple requests
- final channel-complete product UX
- full multimodal ingestion and delivery

## Required Implementation Steps

1. Define policy decision point (PDP) interfaces.
2. Define a generic capability contract that can wrap skills, MCP tools, code execution, backend actions, and delegated agent actions.
3. Add an `IntentResolver` that extracts goal, constraints, modality, and security context from normalized envelopes.
4. Introduce an explicit `Plan` model with:
   - step list
   - candidate capabilities
   - selected capability
   - budget/timeout
   - fallback set
   - status and rationale
   - artifact outputs
   - escalation checkpoints
   - approval-needed markers
5. Implement a lightweight `PlanExecutor` that runs plans step-by-step and emits audit/trace events.
6. Enforce policy checks before every capability/sandbox execution.
7. Implement per-profile execution constraints.
8. Add attestation metadata generation and storage hooks.
9. Implement OpenCode-compatible runtime profile contract.
10. Add progressive rollout mode (`monitor`, `enforce`).
11. Implement capability metadata graph (capabilities, cost, latency, trust level, compatibility constraints).
12. Add policy-aware capability selector (allow/deny + budget + compatibility checks + deterministic ranking).
13. Add capability release channels (`stable`, `candidate`, `deprecated`) and pinning support.
14. Capture runtime capability feedback metrics (success/failure/perf/violations) for adaptive ranking.
15. Route only multi-step, high-risk, or policy-sensitive requests through the planner path; keep simple conversational requests on the existing direct loop path.
16. Introduce first-class capability types and metadata for:
   - database query actions
   - authenticated API actions
   - backend operational actions
   - deployment/hosting actions
   - user notification/escalation actions
17. Extend cooperative orchestration so a multi-agent workflow can execute through the explicit planning path with:
   - bounded handoff count
   - per-agent capability restrictions
   - deterministic fallback when an agent/capability is denied
18. Route scheduler triggers into the same plan model when the trigger represents a multi-step or policy-sensitive action.
19. Add artifact/result contracts so code generation, operational actions, and asset creation steps can emit structured outputs for later delivery or deployment.
20. Add explicit operator/user escalation hooks:
   - request approval
   - request missing secret/data
   - notify over supported reply channels

## Required Tests

1. Unit tests:
   - policy decision logic
   - sandbox profile rule mapping
   - intent resolution
   - plan building and validation
   - capability ranking and fallback ordering
   - artifact collection and escalation decision logic
   - scheduled trigger -> plan conversion
2. Security tests:
   - deny-by-default behavior
   - attempted escalation and path traversal
   - forbidden network access
   - forbidden deployment/backend actions
   - missing-secret denial
3. Integration tests:
   - policy + sandbox end-to-end
   - explicit plan execution end-to-end under policy/budget constraints
   - intelligent capability selection end-to-end under policy/budget constraints
   - direct-loop requests remain on the non-planner path
   - DB/API/custom capability execution under policy
   - cooperative multi-agent plan execution with deterministic handoffs
   - scheduler-triggered plan execution
   - artifact-producing plan execution
4. Stress tests:
   - concurrent coding workloads with quotas
   - concurrent plan execution under policy quotas
5. Non-regression:
   - legacy safe paths still work under compatible mode

## Exit Criteria

1. No policy bypass in security test suite.
2. Sandbox constraints enforced by profile.
3. OpenCode profile executes expected workflows under constraints.
4. Capability selection is policy-safe and deterministic under declared constraints.
5. Explicit plans are auditable, replayable, and do not replace the direct loop for simple requests.
6. Custom skill classes required by use case 2 are representable and executable through the capability model under policy.
7. Cooperative multi-agent execution required by use case 5 can run through the planner path without undefined handoff behavior.
8. Scheduled executions required by use case 3 can be routed through the same governed plan runtime.
9. Asset/deployment actions required by use case 6 are modeled as governed capabilities, even if final product recipes land in M7.
