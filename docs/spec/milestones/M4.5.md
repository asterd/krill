# M4.5 - Runtime Kernel Refactor

## Status

- Priority: `P0`
- Depends on: M0-M4
- Blocks: M5

## Objective

Refactor the Krill runtime kernel to provide a truly lightweight, high-throughput, and failure-resilient foundation before introducing first-class planning and enterprise governance.

This milestone is intentionally allowed to be disruptive. Backward compatibility is not the primary concern; runtime correctness, simplicity, performance, and architectural headroom are.

## Why This Milestone Exists

Current runtime limitations make M5/M6 materially riskier if left unchanged:

1. The in-process bus contract is too weak for a serious orchestration runtime.
2. Hot-path conversation execution is coupled to durable persistence.
3. Session/versioned context primitives are implemented with coarse-grained locking and whole-state persistence.
4. Execution is modeled as one long-lived loop per client instead of a lightweight shared runtime kernel.
5. Telemetry instrumentation is too deep in the hot path for the desired efficiency target.

M4.5 exists to fix the kernel before layering planning, policy, and control-plane functionality on top.

## In Scope

1. New event backbone contract:
   - real fan-out subscriptions
   - explicit backpressure semantics
   - no silent message loss by default
   - support for local runtime mode and external adapter alignment
2. Runtime state-plane split:
   - hot conversation buffer for active execution
   - durable event store for replay/audit
   - optional snapshot store for fast restore
3. Session service refactor:
   - event-sourced lifecycle/history model
   - direct indexes for active session lookup
   - branch/merge/commit/checkpoint as durable event primitives
4. Execution model refactor:
   - remove one-idle-goroutine-per-client as the default model
   - shared worker execution pool
   - lightweight per-thread or per-session serialization only where required
5. Telemetry contract simplification:
   - keep high-value spans on meaningful boundaries
   - remove low-value instrumentation from the hottest paths
6. Memory/history API redesign:
   - explicit errors
   - batch/snapshot primitives where needed
   - separation of runtime buffer access from durable storage concerns
7. Deterministic recovery and failure semantics for the refactored kernel.

## Out of Scope

- intelligent planner logic
- policy-aware capability selection
- control plane APIs
- full distributed consensus or cluster-native execution scheduler
- replacing all protocols with a new transport layer

## Required Architecture Outcomes

1. The bus/event backbone must become a real delivery substrate, not a shared channel shortcut.
2. Hot execution must not require synchronous durable reads/writes on every agent turn.
3. Sessions and versioned context must be replayable and indexed without whole-file rewrites.
4. The runtime must scale by shared execution resources, not by parking goroutines per connected client.
5. Observability must remain strong while materially reducing hot-path overhead.

## Required Implementation Steps

1. Define a new internal event backbone contract with:
   - publish
   - subscribe
   - subscriber isolation
   - backpressure/failure behavior
   - acknowledgement semantics where needed
2. Replace the current local bus implementation with a correct in-memory backbone that mirrors external adapter semantics closely enough for deterministic testing.
3. Introduce a dedicated hot-path conversation buffer abstraction for active threads/sessions.
4. Introduce a durable append-oriented event store abstraction for:
   - message history
   - session lifecycle events
   - branch/merge/commit/checkpoint events
5. Add snapshot primitives for fast restore of long-running sessions.
6. Refactor the session service to use event sourcing plus indexed lookup instead of whole-state JSON persistence.
7. Redesign memory/history interfaces so runtime code can distinguish:
   - hot buffer read/write
   - durable append
   - restore/snapshot
   - retention/trim
8. Replace the default per-client loop ownership model with a shared execution runtime:
   - lightweight work scheduling
   - explicit serialization boundaries per thread/session when needed
   - bounded queues with observable overflow behavior
9. Reduce low-value telemetry inside the hottest code paths and keep tracing at meaningful execution boundaries only.
10. Add benchmark suites for:
   - inbound throughput
   - reply throughput
   - session restore latency
   - concurrent active thread execution
   - durable append overhead
11. Add failure-injection coverage for:
   - backpressure
   - storage write failure
   - recovery after restart
   - duplicate delivery
   - cancellation during execution

## Required Tests

1. Unit tests:
   - event backbone semantics
   - conversation buffer semantics
   - event store append/replay semantics
   - snapshot/restore logic
   - session index correctness
2. Integration tests:
   - inbound -> execution -> reply over the new kernel path
   - restore after restart with active persistent sessions
   - branch/merge/commit replay correctness on the new session model
3. Failure-injection tests:
   - bounded queue overflow behavior
   - storage append failures
   - duplicate message delivery
   - cancellation during active execution
   - restart with partially persisted state
4. Benchmark tests:
   - local event backbone throughput
   - active thread scheduling throughput
   - restore latency under realistic session counts
   - telemetry overhead comparison before/after refactor
5. Non-regression:
   - protocols still produce correct request/reply behavior on the new runtime kernel
   - scheduler/session features from M4 still function on the new substrate

## Exit Criteria

1. The new event backbone has explicit and tested delivery semantics with no silent drop in the default runtime mode.
2. Active execution no longer depends on synchronous durable history reconstruction on every turn.
3. Session/versioned context operations are event-backed, indexed, and replayable.
4. The runtime no longer relies on one idle loop goroutine per client as the default scaling model.
5. Kernel benchmarks show materially improved throughput and/or tail-latency versus the pre-refactor runtime.
6. M5 can be implemented on top of the new kernel without reopening core execution, session, or bus contracts.
